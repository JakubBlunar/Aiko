# Agent Instructions

## Project Overview

Aiko is a web-based AI companion built around:

- **Python 3.11+** backend (FastAPI + WebSocket) under `app/`. Entry point: `python -m app.web` (or the `aiko-web` console script).
- **React + Vite + PixiJS** frontend under `web/` (Live2D avatar, chat, voice controls, settings drawer, document upload).
- **Ollama** for chat (via `OllamaClient` directly, not LangChain). The `chat_llm` block can route to any OpenAI-compatible endpoint instead.
- **RealtimeSTT** + **Pocket-TTS** for voice in/out, with **client-owned audio I/O**: the browser / Tauri shell captures the microphone (48 kHz Int16 mono, browser DSP) and plays back TTS, streaming raw PCM frames over the existing WebSocket. See [`docs/voice-mode.md`](docs/voice-mode.md) for the binary frame protocol and the voice-ownership lock used when multiple windows are open.
- **LanceDB** for vector RAG over memories, recent chat messages, and uploaded documents.
- **SQLite** (`data/chat_sessions.db`) as the source of truth for messages, summaries, and memory metadata.

There is no desktop / Qt / LangChain code. The web UI is the only UI.

## Embedded MCP Server

The app exposes an MCP server on `http://localhost:6274/sse` for development tooling. Start the app first, then connect any MCP client.

### Cursor Setup

Already configured in `.cursor/mcp.json`. The tools appear as native MCP tools — call them directly, no wrapper scripts needed.

### VSCode / Copilot Setup

Add to your MCP settings (`.vscode/mcp.json` or user settings):

```json
{
  "servers": {
    "assistant": {
      "type": "sse",
      "url": "http://localhost:6274/sse"
    }
  }
}
```

### Tools

| Tool | Args | Returns |
|------|------|---------|
| `send_message` | `message: str`, `skip_tts: bool = false` | Assistant response text. The web UI updates live as the message streams. |
| `get_status` | — | JSON: model name, context window, TTS engine, agent tool count, recent metrics. |
| `list_agent_tools` | — | JSON array of `{name, description}` for every agent tool currently registered. |
| `get_last_response_detail` | — | JSON timing breakdown for the last turn (`llm_ms`, `tts_ms`, etc.). |
| `clear_history` | — | Clears the active session in `chat_sessions.db`. |

### Resources

| URI | Content |
|-----|---------|
| `assistant://history` | Recent conversation messages (JSON). |
| `assistant://config` | Current settings snapshot (JSON). |

### Debugging Workflow

1. **Confirm connection**: Call `get_status` — verify `model`, `tool_count`, and `tts.engine`.
2. **Test agent**: Call `send_message` with `skip_tts: true` to avoid audio playback during automated testing.
3. **Check timing**: Call `get_last_response_detail` — `llm_ms` is the model time, `tts_ms` is speech synthesis time.
4. **Read logs**: The app console prints tool registry rebuilds, `TurnRunner` two-pass execution, and proactive nudges at INFO level.

### Adding Custom MCP Tools

If the existing tools are not enough for your debugging scenario, add new ones directly in `app/mcp/server.py`. The server has full access to `SessionController` and all internal state.

```python
@mcp.tool()
def my_debug_tool(some_arg: str) -> str:
    """Description of what this tool does."""
    # Access any internal state via the `session` reference:
    #   session._settings, session._chat_db, session._memory_store,
    #   session._rag_store, session._tool_registry, etc.
    # The app must be restarted for new tools to take effect.
    return "result"
```

You are encouraged to add any MCP tool you need to debug a problem. Common examples: inspecting agent message history mid-turn, dumping the system prompt, reading TTS queue state, checking embedding search results, or triggering specific `SessionController` methods. After adding a tool, restart the app and it will appear automatically.

### Architecture Notes

- `app/mcp/server.py` — FastMCP server definition with all tools/resources. Add new tools here.
- `app/mcp/runner.py` — Runs uvicorn in a daemon thread; stops on app shutdown.
- `app/core/session/session_controller.py` — Starts the MCP server in `__init__`, stops in `shutdown()`. Message listeners notify the web UI of MCP-triggered messages over WebSocket.
- Config: `config/default.json` key `mcp_server` (`enabled`, `port`).

## Code Conventions

- Python 3.11+, dataclasses + slots for settings, FastAPI for the HTTP/WebSocket layer.
- LLM calls go through `app/llm/ollama_client.py` (`chat`, `chat_stream`, `chat_with_tools`). Tool dispatch happens in `app/core/session/turn_runner.py` via a pre-stream `chat_with_tools` pass; the streaming reply runs immediately after.
- The agent's available tools are built once per turn from `config.tools` by `SessionController.rebuild_tool_registry()` — see `app/llm/tools/builtins.py` for the live set (`get_time`, `recall`, `web_search`).
- TTS text processing (`prepare_tts_text` in `app/core/session/session_text_utils.py`) applies only to the spoken stream, not the chat transcript.
- SQLite (`data/chat_sessions.db`) is the source of truth; LanceDB (`data/lancedb/`) mirrors `memories`, asynchronously indexes `messages`, and holds chunked uploaded `documents`.
- Inline tags Aiko emits: `[[reaction:...]]` (mood label, drives Live2D expression and TTS reaction-to-speed); `[[remember:...]]` / `[[remember:self:...]]` (writes a long-term memory); `[[arc:...]]` (H1 conversation-arc self-tag — one of `casual_check_in / support / planning / reflection / playful / silly`, lands at confidence `0.85` on the regex 0.5 < self-tag 0.85 < smoother 0.95 ladder, see [`app/core/conversation/conversation_arc.py`](app/core/conversation/conversation_arc.py)); and `[[prosody:...]]` (per-sentence vocal delivery — one of `whisper / soft / slow / fast / firm`, orthogonal to the reaction mood label, applies to the single sentence that opens with the tag, see [`app/core/voice/cadence.py`](app/core/voice/cadence.py)). All four are stripped from the spoken/chat output before TTS / transcript persistence. Stage-direction earcons (`[[laugh]] [[sigh]] [[gasp]] [[hum]] [[tsk]] [[chuckle]] [[soft_sigh]] [[sharp_gasp]] [[breath]] [[mm]]`) are also inline tags but route through a side-channel earcon player rather than TTS.
- Persona file at [`data/persona/aiko_companion.txt`](data/persona/aiko_companion.txt) is user-editable; the "Reading {user_name}" section instructs Aiko to mirror the user's register from the live `vocal_tone` and `user_state` prompt cues. The same nudge is mirrored in `_SPEECH_GRAMMAR_ADDENDUM` ([`app/core/session/prompt_assembler.py`](app/core/session/prompt_assembler.py)) so a deleted/rewritten persona file doesn't lose the behaviour. **Every "Jacob" in the persona has been templated** as the literal placeholder `{user_name}` (Phase 4d). `PromptAssembler._load_persona()` calls `.format(user_name=resolve_user_display_name(...))` once per load; new persona text must keep all braces as `{user_name}` (any other `{...}` token crashes `.format()` and falls back to the raw text via the try/except guard). The legacy backup file `aiko_companion_backup.txt` was deleted — never restore it. **Tool source-of-truth split**: the persona does NOT carry a per-tool catalogue. Every agent tool is already passed to the model natively via the `tools=` parameter (`TurnRunner` -> `chat_with_tools`), and each tool's `schema()` description ([`app/llm/tools/*.py`](app/llm/tools/) — `builtins`, `world`, `goals`, `file_tasks`) is the imperative, self-contained "what this tool does / when to call it / sync vs async" reference. The persona's "How tools work" block only describes cross-cutting *handling* rules (sync returns inline, async returns a task id whose result lands a later turn, the anti-deferral rule, ask-back-on-disambiguation, never paste raw output). **When adding a tool, put the description in its `schema()` — do NOT re-document it in the persona.** The persona was trimmed ~47% (Phase: persona-trim-tool-budget) by removing this duplication and compressing each K-feature cue section to its essential anti-pattern; the literal `[[...]]` tag vocabulary Aiko must emit (reaction taxonomy, prosody values, `[[remember:...]]` / `[[gap:...]]` / `[[predict:...]]` / `[[arc:...]]` / `[[goal:...]]` / `[[touch:...]]` / `[[conflict:...]]`) was preserved verbatim.
- **Tool under-calling fix — forced choice + escape tool** (NOT reasoning effort). Chatty models (gpt-5-mini especially) lose the implicit "text vs tool" decision on `tool_choice="auto"`: they narrate intent ("I'll list the folders") instead of emitting the call. **`reasoning_effort` is NOT the lever** — minimal / low / medium all defer identically; medium just adds ~8s of latency, so the OpenAI-compat client keeps `reasoning_effort="minimal"` on *every* surface ([`app/llm/openai_compatible_client.py`](app/llm/openai_compatible_client.py)). The actual fix lives in `TurnRunner._maybe_run_tool_pass` ([`app/core/session/turn_runner.py`](app/core/session/turn_runner.py)): it forces `tool_choice="required"` and appends a synthetic `respond_directly` escape tool (`_RESPOND_DIRECTLY_SCHEMA`, never registered in the `ToolRegistry`). This reframes the choice from "text vs tool" (which the conversational prior always wins) to "which tool" (where `list_file_roots` clearly wins for "what files can you see?"). The pass filters the escape tool out of `response.tool_calls`: if it's the only pick the model is signalling "just answer" and the pass returns WITHOUT appending the tool-calls message (a dangling tool_call with no result would 400 the streaming pass); if a real tool is also picked, only the real one is dispatched. `tool_choice` is a new keyword-only param on `chat_with_tools` across the `ChatClient` protocol + both clients — forwarded to OpenAI-compatible providers, ignored (harmless) by Ollama's Go server. The tool pass caps `num_predict = min(max_tokens, 512)`. Tests: `tests/test_turn_runner_tool_pass.py` (`ForcedChoiceEscapeToolTests` + budget). **If the model still won't call a real tool, the next lever is a focused tool-router system prompt on the decision pass (option B) — NOT raising reasoning effort.** **P14 gate**: the pass no longer runs on *every* turn — a pure heuristic gate ([`app/core/session/tool_pass_gate.py`](app/core/session/tool_pass_gate.py)) skips it on turns with no tool-shaped signal and no continuity signal (finished-task block / active task / previous turn dispatched a tool / MCP force flag). When the pass does run, the forced-choice + escape-tool semantics above are untouched. **When adding a new tool, add its name to `_TOOL_FAMILY` (and patterns if it's a new family) in `tool_pass_gate.py`** — unmapped tools force the gate into always-run (safe but pays the full TTFT tax again). Kill-switch: `agent.tool_pass_gate_enabled=false`. Debug: MCP `get_tool_gate_state` / `force_tool_pass`, `tail_logs(module_contains="tool_pass_gate")`.

- **File size guidance** — Cursor's indexer slows down on very large source files (we hit it at ~5k lines on `SettingsDrawer.tsx` and `session_controller.py`). Keep Python files below ~1,500 lines and React/TS components below ~1,000 where you can; if a file exceeds ~2,500 lines, schedule a split before adding more. Established split patterns:
    - **Python**: feature mixins under `app/core/<area>/*_mixin.py`. State ownership stays on the host class (`SessionController.__init__` keeps every `self.*` assignment); mixins only group methods. Reference: [`app/core/session/avatar_mixin.py`](app/core/session/avatar_mixin.py), [`app/core/session/inner_life_providers_mixin.py`](app/core/session/inner_life_providers_mixin.py), [`app/core/session/speaking_window_jobs_mixin.py`](app/core/session/speaking_window_jobs_mixin.py), [`app/core/session/post_turn_mixin.py`](app/core/session/post_turn_mixin.py). Tests that patch `app.core.session.session_controller.<symbol>` for a method that has since moved must patch the **mixin module** instead — the patch must target the module where the symbol is *looked up*.
    - **React/TS**: extract panels and tabs into a feature folder under `web/src/components/<feature>/*.tsx`. Use explicit `Props` interfaces, not parent closures, so the new file is self-contained and source-testable. Reference: [`web/src/components/settings/`](web/src/components/settings/) (Voice, Avatar, Memory + 5 sub-panels, World, Together, Diagnostics, Identity each in their own file).
    - **Where to put new code** — when extending an inner-life prompt provider, post-turn hook, or speaking-window job, place the new method on the relevant mixin (not on `session_controller.py` directly). When adding a new Settings drawer surface, create a new file under `web/src/components/settings/` with its own `Props` rather than another inline tab block.
- The Settings drawer has a dedicated **Memory** tab ([`web/src/components/SettingsDrawer.tsx`](web/src/components/SettingsDrawer.tsx)) for inspecting + editing long-term memories. It paginates server-side via `GET /api/memories?limit=&offset=&kind=` (page size 50, response includes `total` + `cap`), with edit-in-place, manual create (`POST /api/memories`, dedupes via the existing cosine-collapse path and toasts "merged into memory #N"), salience editing, kind filter / sort selectors, and a **pin** toggle (`POST /api/memories/{id}/pin`). Pinned rows are **immune to `MemoryStore.decay()` and never selected as `prune()` victims**, and `RagRetriever` adds a `+0.05` score bonus for pinned hits. Pinning lives in SQLite (`memories.pinned`, schema v5) only — the LanceDB mirror is intentionally not aware of it, to avoid a destructive vector-store rebuild on existing user databases. Frontend Zustand state is `memoryView: { items, total, cap, page, pageSize, kindFilter, order }` plus `applyMemoryAdded` / `applyMemoryUpdated` / `applyMemoryDeleted` reducers; `memory_added` only prepends to the visible page when on page 0 with `order=recent` and a matching kind filter (otherwise just bumps `total`). New WS event: `memory_updated`. The default `memory.max_memories` cap was bumped from 500 to 5000 — search remains a sub-millisecond NumPy pass at that size and the LanceDB-backed retrieval is sub-linear, so the headroom is essentially free.

The Zustand reducer `applyWorldPatch` merges them in place so the new **World** tab in `SettingsDrawer.tsx` stays live. **"Give Aiko a cookie" is intentionally silent** — items appear in her room with `given_by="user"` and Aiko notices them on her next turn through the prompt block, no proactive message is fired. Consumable items (food) decrement on `consume_item` and the row is deleted at quantity zero; non-consumables clamp at zero so the lamp stays in the room.

- **Prompt-cache prefix-stability ladder** — `PromptAssembler.assemble_with_budget` lays `system_parts` out from most-stable (T0) to most-volatile (T6) so OpenAI's prompt cache hits the longest possible prefix on every turn. The contract is encoded twice — once as the audit constant [`_PROMPT_BLOCK_TIERS`](app/core/session/prompt_assembler.py) near the top of the file, and once as the literal `if block: system_parts.append(block)` cascade further down. The tiers, with one-line examples: **T0 stable** (persona, grammar addenda, profile, petname, catchphrase) → **T1 semi-stable** (relationship, axes, arc, agenda, goals, day_color, anniversary) → **T2 summary** (compaction-only) → **T3 RAG memory** → **T4 ambient** (grounding line, world, activity, circadian, pajama, ambient noise, sensory anchor) → **T5 affect / style** (affect, mood hint, mood shell, style signal, user state, vocal tone) → **T6 detectors** (everything that takes `user_text` as input: novelty, stagnation, clarification, belief gaps, knowledge gaps, opinion injection, …). **When adding a new prompt block, pick the tier that matches the block's lifetime, append it inside that tier's `if` cluster, and add its name to `_PROMPT_BLOCK_TIERS`.** Cross-tier inversions are caught by `tests/test_prompt_assembler.py::PromptCachePrefixOrderingTests` (persona < affect, profile < memory, summary < memory, memory < detectors, …). Observability: `ChatUsage.cached_tokens` is lifted from `prompt_tokens_details.cached_tokens` on every OpenAI response, surfaces on the `turn done:` INFO line as `cached=N cached_pct=%.1f`, and is mirrored on MCP `get_last_response_detail`. Ollama and most non-OpenAI providers leave the field at `0`, which is the correct null answer. Workers share the same `ChatClient` but write to a *different* cache slot (their `messages[0]` is not Aiko's chat system prompt), so worker calls do not invalidate the chat cache nor share its discount. Full design + worked pricing example: [`docs/prompt-caching.md`](docs/prompt-caching.md).

- **Memory tiers + revival drift + IdleWorker framework (schema v8)** — Long-term memory is no longer a flat pool. Every row in [`memories`](app/core/memory/memory_store.py) now carries a `tier` (`scratchpad` / `long_term` / `archive`) and a `revival_score` in `[0, 1]`. Auto-extracted observations (`MemoryExtractor`, `ReflectionWorker`, `DreamWorker`) land in `scratchpad`; verified anchors (`PromiseExtractor`, `CatchphraseMiner`, `SharedMoments`, `RelationshipPulse`, `[[remember:...]]` tags via `TurnRunner`, manual REST/UI adds, milestone memories) write directly to `long_term`; pinned rows are always coerced to `long_term`. `MemoryStore.decay` is now **wall-clock-driven** (proportional to elapsed time since `memory.last_decay_run_at` in the new `kv_meta` key-value table, clamped by `decay_max_catchup_days`) with per-tier rates and a revival rebate (`salience += revival_coefficient * elapsed * revival_score`). `prune()` enforces per-tier caps independently. **Revival detection** (`SessionController._mark_revived_memories`) runs post-turn: it reads the surfaced-IDs snapshot off `RagRetriever.last_surfaced_memory_ids` and bumps `revival_score` on memories whose content shares ≥ `revival_min_word_overlap` (default 3) content words with Aiko's reply. A new **`IdleWorkerScheduler`** (`app/core/proactive/idle_worker_scheduler.py`) runs background workers during quiet windows (gated by `SessionController._is_user_idle`: no Live mode, no turn in flight, no recent user activity); it caps one worker per tick to avoid stacking and persists `last_run_at` per worker to `kv_meta`. - **Surprise / novelty detector (K6)** — Per-turn signal that lets Aiko react with real surprise when Jacob pivots away from the recent topic baseline. No schema change; the detector is purely in-process. The [`NoveltyDetector`](app/core/conversation/novelty_detector.py) keeps an in-memory `collections.deque[np.ndarray]` of size `novelty_window=12` on each `SessionController`, lazily warmed once per session from [`RagStore.list_recent_user_vectors`](app/core/rag/rag_store.py) filtered to `role='user'` rows whose `session_id` starts with the current user prefix (so a topic discussed yesterday won't re-fire "this is new" today). On every turn it embeds `user_text` via the shared `Embedder`, computes `distance = 1 - cosine(vec, centroid)` against the renormalised mean of the ring, classifies into `strong_novelty` (≥ 0.55), `mild_shift` (≥ 0.35), or silent, and appends the new vector to the ring so the baseline keeps moving. A `novelty_cooldown_turns=2` suppression window prevents back-to-back "you keep saying surprising things" beats; cold-start installs stay silent until the ring fills past `novelty_warmup_min=3`. The signal surfaces through a new `novelty` inner-life provider on [`PromptAssembler`](app/core/session/prompt_assembler.py) (same shape as F2 `knowledge_gaps`: takes the live `user_text`, lands right after `belief_gaps_block`, dropped under `aggressive=True`). The detector also exposes `last_distance` / `last_band` attributes (additive, no behaviour change for K6 callers) so the K18 sibling can read the per-turn measurement without re-embedding. **MCP-debuggable**: the detector module logs one INFO line per turn — `novelty-detector: distance=%.3f band=%s window=%d user=%s` — plus a one-shot `novelty-detector: warmed ring=N user=X` on first detect; both grep-friendly via `tail_logs(module_contains="novelty")`. To repro a hit end-to-end: send three on-topic messages (e.g. about debugging Python), then one wildly off-topic (e.g. about bee-keeping), and watch the next inner-life block include "Heads-up: Jacob just brought up something well outside the recent baseline." Tests: `tests/test_novelty_detector.py`, plus extensions to `tests/test_prompt_assembler.py` and `tests/test_rag_store.py`.

- **Topic stagnation detector (K18)** — Sibling of K6 that fires on the *inverse* signal: when the rolling per-turn distance to the K6 centroid stays low across a window (default 6 turns), Aiko gets a "Heads-up: you've been circling the same topic with Jacob for a bit" / "Heads-up: this thread has been pretty looped for a while" cue and is nudged toward a soft pivot, deepening, or a real off-ramp. Bands the rolling mean: `mean < stagnation_strong_threshold=0.10` → `strong_lull`, `mean < stagnation_mild_threshold=0.18` → `mild_lull`, otherwise silent. Three suppression gates keep the cue rare: warmup until the deque is full, `stagnation_cooldown_turns=4` after each fire (longer than K6's because lulls are by nature drawn-out), and a `stagnation_post_novelty_suppression_turns=3` window right after K6 fires so a fresh topic shift doesn't immediately read as "we've been on this for a while". Pure module [`app/core/affect/vulnerability_budget.py`](app/core/affect/vulnerability_budget.py) owns the math: `BudgetState{spent, last_decay_at}` dataclass + `classify_disclosure_tier` heuristic (priority ladder: Aiko's `tier=N:` self-tag fast-path -> tier-3 marker patterns -> tier-2 marker patterns -> length-based lift >= 100 chars -> default tier 1), with `tier_cost(tier)` mapping to 1 / 3 / 6 tokens by default. Capacity is `compute_capacity(closeness, trust)` — linear interp over `(closeness+trust)/2` from `[-1, 1]` to `[min_cap=1, max_cap=12]`, both axes sourced from [`RelationshipAxesStore`](app/core/relationship/relationship_axes.py); a brand-new user with no relationship state defaults to neutral -> midpoint capacity ~6. Bucket regenerates over wall-clock hours (`apply_decay`: `new_spent = max(0, spent - regen_per_hour * elapsed_h)`, default `0.5 tokens/hour` so a tier-3 spend recovers in ~12h and a full capacity bucket in ~24h). Storage is **on `kv_meta`** — one JSON key `aiko.vulnerability_budget` carrying `{spent, last_decay_at}`, same `aiko.*` namespace as K27. **Soft enforcement only**: the provider's rendered cue is the *only* mechanism — never suppresses the underlying memory write, never filters reply text, never caps the spend at capacity (going over produces a *stronger* cue next turn). [`PostTurnMixin._post_turn_inner_life`](app/core/session/post_turn_mixin.py) re-parses `[[remember:self:...]]` tags via the `_SELF_TAG_RE` mirror of `_REMEMBER_TAG_RE`, delegates to `vulnerability_budget.compute_spend_for_self_tags` (returns a `SpendReport`), and persists the new state — best-effort, logs `vulnerability-budget spend: cost=N tier_counts={1:N,2:N,3:N} spent=Y -> Z` on every fire. [`InnerLifeProvidersMixin._render_vulnerability_budget_block`](app/core/session/inner_life_providers_mixin.py) reads + decays + persists-back + renders on every turn; four cue shapes (half-spent / at-cap / over-cap / **low-ceiling override** when `capacity <= 2` AND `spent > 0`). The block lands in `system_parts` **immediately after** `self_noticing_block` so the "register I'm in / how much have I shared" pair reads as one self-aware family; **NOT dropped under `aggressive=True`** (a tight budget is exactly when the over-cap warning matters most — long replies compound over-disclosure); **NOT in the K16 grounding-line suppression matrix** because it's a pacing cue, not an ambient grounding block. - **External MCP-server clients — background-lane tools (Phase 1)** — The app is now an MCP **client** as well as a server. Where [`app/mcp/server.py`](app/mcp/server.py) *exposes* a debug server, the new [`app/mcp/client/`](app/mcp/client/) package *connects out* to external MCP servers (official filesystem server, browser-automation server, …) and consumes their tools. **Lane rule (hard):** discovered MCP tools register **only** into the background-worker `WorkflowSkillRegistry` (the goal-workflow planner lane) — **never** into the brain's fast `ToolRegistry`. - **Brain Orchestration Phase 2 — scaling foundations (schema v17)** — Phase 2 layers four new top-level concepts on top of the phase-1 `BrainEventQueue` / `BrainLoop` / `tasks` table without breaking any wire-level contract. (1) An **append-only task event log** ([`task_events`](app/core/tasks/task_events.py) table + [`TaskEventStore`](app/core/tasks/task_events.py) with 11 stable `EVENT_*` constants — `started`, `progress`, `phase_change`, `input_question`, `input_answer`, `completed`, `failed`, `cancelled`, `interrupted`, `heartbeat_stalled`, `child_spawned`) records every meaningful lifecycle moment + handler-defined custom events via the new `TaskEventEmit(type, data)` outcome. (2) A **dedicated input/answer history** ([`task_inputs`](app/core/tasks/task_inputs.py) table + [`TaskInputStore`](app/core/tasks/task_inputs.py) with four status constants — `pending`, `answered`, `superseded`, `cancelled`) unblocks multi-clarification flows ("which folder?" → "which file?" → "which line?") — the orchestrator calls `supersede_pending_for_task` before every fresh `TaskInputNeeded` so no orphan `pending` row leaks. The phase-1 single-slot `tasks.input_request` is still kept in sync as a denormalised "what's the current question?" mirror for cheap WS / REST reads. (3) **`phase` / `parent_task_id` / `heartbeat_at` as first-class `tasks` columns** — `phase` is the human-readable label (`"extracting_prices"`) the LLM and the TasksTab read instead of trying to make `progress=0.42` meaningful for a discrete state machine; `parent_task_id` records spawn relationships as a single-parent tree (NOT a DAG — keep it simple) so cancel-cascade and the future "wait on all children" feature can both walk it; `heartbeat_at` is bumped on every emit so a daemon thread can flag "running but silent for 5 minutes" without false positives. (4) **Heartbeat sweep + cleanup worker — two independent daemons.** [`HeartbeatChecker`](app/core/tasks/task_heartbeat.py) is a `daemon=True` thread that wakes every `task_heartbeat_check_interval_seconds=30` (floor `5`, set to `0` to disable) and calls `TaskStore.list_stalled` for `running` rows older than `task_stalled_seconds=300` (floor `60`); action `warn` (default) just logs + appends `EVENT_HEARTBEAT_STALLED`, action `fail` also flips to `status='failed'`, action `disabled` is the kill switch. `awaiting_input` / `paused` are never flagged — they're stalled *by design*. [`TaskCleanupWorker`](app/core/tasks/task_cleanup_worker.py) is a regular `IdleWorker` that runs in quiet windows, deletes terminal rows older than `task_cleanup_retention_days=30` (floor `1`, capped by `task_cleanup_max_rows_per_tick`), and cascade-deletes the matching `task_events` + `task_inputs` rows in order. `TaskOrchestrator.cancel(task_id)` also walks `parent_task_id` children depth-first when `task_cascade_cancel_children=true` (default), appending `EVENT_CANCELLED` with `data={"by_cascade": true}` on every descendant. **REST**: `GET /api/tasks/{id}/events?limit=&offset=&type=` (paginated chronological log) and `GET /api/tasks/{id}/inputs` (full input history including superseded/cancelled rows); `GET /api/tasks/{id}` now includes `phase` / `parent_task_id` / `heartbeat_at`. **WS**: `task_progress` payloads now carry `phase` when present. **Frontend**: `TaskStrip.tsx`'s `TaskChip` renders `phase` next to the status; `TasksTab.tsx` adds a lazy `EventsExpander` per row so a 10K-task history doesn't materialise upfront. **Six new agent settings** (defaults / floors): `task_heartbeat_check_interval_seconds=30/5`, `task_stalled_seconds=300/60`, `task_stalled_action="warn"` (unknown values fall back to `warn`), `task_cascade_cancel_children=true`, `task_cleanup_retention_days=30/1`, `task_cleanup_interval_seconds=21600/600`. **MCP debug surface**: `get_heartbeat_state()`, `force_heartbeat_sweep()`, `force_run_idle_worker("task_cleanup")`, `list_task_events(task_id)`, `list_task_inputs(task_id)`, `emit_task_event(task_id, type, data)`. **Recovery interaction**: `recover_interrupted_tasks` now also calls `cancel_pending_for_task` on every row demoted from `running` to `interrupted` so the user never sees a pending question for a task they thought was done; `awaiting_input` rows are preserved as-is (pending input stays valid). Full design + symptom-to-grep table: [`docs/brain-orchestration.md#phase-2--schema-v17-scaling-foundations`](docs/brain-orchestration.md#phase-2--schema-v17-scaling-foundations).

## Debugging via logs

Aiko writes a single, level-disciplined log stream that lands in three places at once: stderr (live console), `data/app.log` (rotating, 5 MB × 5 files), and an in-process ring buffer (last 1000 lines) exposed over MCP. Every line is shaped the same way — memorise the shape and the canonical fields and you can answer most "what just happened?" questions with one or two MCP calls.

### a. Where to look (priority order)

1. **`tail_logs(n=200, level="DEBUG", module_contains=…)`** — first stop. Instant, scoped, level-filterable. Use `module_contains="prompt"` to focus on `app.core.session.prompt_assembler`, `module_contains="ollama"` for the LLM client, etc.
2. **`read_log_file(lines=2000, grep="turn=abc12345")`** — when the issue is older than the ring (~1000 lines back) or you want to follow one turn end-to-end. Hits the rotating `data/app.log` (and its `.1`…`.5` siblings).
3. **`data/app.log` directly via Read tool** — for offline / cross-session forensic work, including reading rolled siblings.
4. **`terminals/*.txt`** — last resort for noise outside the `app.*` loggers (uvicorn, fastmcp, library warnings).

`get_log_config` returns the current effective configuration (level, file path, module overrides, ring usage).

### b. Standard line shape

Every record is formatted as:

```
[YYYY-MM-DD HH:MM:SS,mmm] LEVEL [logger.name turn=abc12345] message text key1=val1 key2=val2
```

`turn=…` is an 8-char hex correlation id allocated by `TurnRunner.run()` and propagated via a `contextvars.ContextVar` ([`app/core/infra/log_context.py`](app/core/infra/log_context.py)). When no turn is active you'll see `turn=-`. The canonical structured fields (memorise these for precise greps):

- **Turn lifecycle**: `model=`, `ctx_pct=`, `prompt=`, `completion=`, `first_token_ms=`, `total_ms=`, `eval_ms=`, `tools=`, `compactions=`, `mood=`, `aborted=`, `chars=`, `filler=`.
- **Prompt assembly**: `ctx=`, `budget=`, `est_tokens=`, `sys=`, `hist=`, `user=`, `rag_tokens=`, `history_msgs_in=`, `history_msgs_out=`, `inner_blocks=`, `summary_active=`, `compaction=`, `aggressive=`.
- **Scheduler**: `jobs_run=`, `elapsed_ms=`, `queue_after=`, `idle=`, `names=`, `submitted=`, `ran=`, `cancelled=`, `windows_opened=`.
- **Ollama**: `msgs=`, `tools=`, `stream=`, `prompt_tokens=`, `completion_tokens=`, `tool_calls=`, `format_json=`, `stopped=`.
- **STT**: `chars=`, `duration_ms=`, `model=`, `language=`, `init_ms=`, `device=`, `sample_rate=`, `max_s=`, `silence_s=`.
- **TTS**: `voice=`, `provider=`, `init_ms=`, `chunk_chars=`, `speed=`, `generate_ms=`, `played_ms=`, `queue_depth=`, `drained_chunks=`.
- **Proactive**: `source=` (`prepared`|`llm`).

### d. Level cheat sheet

The level dial is a contract — every new log call must respect it.

- **ERROR** — only real failures (turn lost, engine crashed, non-2xx). Always shown.
- **WARNING** — degraded paths and retries (model warmup failed, voice switch failed, scheduler job raised but loop continued).
- **INFO** (default) — one structured line per turn (`turn done:`), lifecycle moments (boot, `ollama connected:`, `STT engine ready:`, `TTS engine ready:`, `TTS voice switched:`, `STT capture done:`, scheduler init/shutdown, compaction triggered, proactive spoken).
- **DEBUG** — the tweaking firehose (per-Ollama-call timings, scheduler per-drain summaries, prompt-assembler retrieval breakdown, STT capture lifecycle, TTS enqueue/play, RAG cache hits, full transcript previews).

Practical presets:

| Goal | How |
|---|---|
| Default | `level=INFO` in config. Turn-by-turn trace + lifecycle. |
| Investigate one module | Keep global `level=INFO`; bump just the relevant module via `set_log_level("app.core.session.prompt_assembler", "DEBUG")` (runtime) or `logging.module_levels` in `config/user.json` (persistent). |
| Production / quiet | `level=WARNING` or `level=ERROR` — a healthy session emits zero log lines. |
| Full firehose | `level=DEBUG` — only when actively tweaking; expect ~hundreds of lines per turn. |

### e. Practical workflow

1. **Before reproducing**: `set_log_level("app.core.<suspect>", "DEBUG")` on the module you suspect.
2. **Trigger the issue**: `send_message(skip_tts=true)` (or use the UI / voice).
3. **Read back**: `tail_logs(module_contains="<suspect>")` for the most recent ring entries, then `read_log_file(grep="turn=<id>")` to widen the lens to the whole turn.
4. **If you need a new field**: open the relevant module, add a structured log line at the right level (use `key=value` so a future grep stays cheap), document the new field in §b above, and restart the app.
